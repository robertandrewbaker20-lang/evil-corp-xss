What is Cross-Site Scripting (XSS)?

Cross-Site Scripting — known as XSS — is one of the most common vulnerabilities on the web. It occurs when a website takes input from a user and displays it back on the page without checking what that input actually is.

Normally, a feedback form takes your name and message and shows them on screen. But what if instead of a name, someone typed actual JavaScript code? If the website isn't careful, the browser will execute that code — not display it as text.

Here is what a basic XSS payload looks like. An attacker types this into a form field instead of normal text:

EXAMPLE PAYLOAD
<script>alert('XSS')</script>

If the website is vulnerable, instead of showing that text on screen, the browser runs it as code and a popup appears. That popup proves the attacker has control over what the browser executes.

Why Is This Dangerous?

A simple alert popup seems harmless. But XSS is dangerous because JavaScript can do much more than show a message. An attacker can use XSS to:

— Read sensitive data stored in the browser
— Steal cookies that keep users logged in
— Redirect users to fake login pages
— Execute actions on behalf of the victim

Cookies are especially valuable. Browsers store small pieces of data called cookies — including session tokens and other sensitive values. JavaScript can read these using document.cookie.

COOKIE THEFT EXAMPLE
<script>alert(document.cookie)</script>

How To Protect Against XSS

The defense against XSS is simple: never trust user input. Before displaying anything a user typed, a safe application converts special characters like < and > into harmless text so the browser never tries to execute them as code.

This process is called input sanitization — and it is the single most important defense against XSS. Evil Corp takes this extremely seriously across all of our platforms.
